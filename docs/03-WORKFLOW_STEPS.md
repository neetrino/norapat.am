## 🤖 AI-ի համար — խստորեն կիրառելի քայլեր

**Յուրաքանչյուր օգտատիրոջ նամակի դեպքում AI-ն պետք է անի հետևյալը հենց այս հերթականությամբ.**

### Փուլ A. Նախքան ցանկացած գործողություն

| # | Քայլ | Պայման | Գործողություն |
|---|------|--------|----------------|
| A1 | Խնդիրը հասկանալ | Միշտ | Կարդալ օգտատիրոջ հաղորդագրությունը ամբողջությամբ։ Պարզել՝ *ինչ* (ֆիչ/ուղղում/հարց), *որտեղ* (էջ, API, ադմին), *արդյունք*։ |
| A2 | Կոնտեքստ | Միշտ | Կարդալ `docs/PROJECT_CONTEXT.md` — կառուցվածք, API, կոնվենցիաներ, որպեսզի **չշեղվել** նախագծից։ |
| A3 | Ֆունկցիոնալ | Եթե խնդիրը **նոր ֆիչ** կամ **էջ** է | Կարդալ `docs/02-FUNCTIONAL.md` — համապատասխան բաժին (Home, Shop, Product, Checkout, Admin…) և ապահովել, որ լուծումը **համապատասխանի** պահանջներին։ |
| A4 | Ճարտարապետություն | Եթե **մեծ փոփոխություն** (նոր API, ինտեգրացիա, մոդել) | Կարդալ `docs/01-ARCHITECTURE.md` — համոզվել, որ լուծումը **fits** ճարտարապետությանը։ |

### Փուլ B. Պլան — մինչև կոդ գրելը

| # | Քայլ | Պայման | Գործողություն |
|---|------|--------|----------------|
| B1 | Պլան | Միշտ, եթե կպահանջվի կոդ | Ձևակերպել **ինչ** կփոխվի/կստեղծվի (ֆայլեր, էջեր, API, Prisma)։ |
| B2 | Կախվածություններ | Եթե մի քանի մաս կա | Նշել **հերթականություն** (օր. նախ Prisma մոդել, ապա API, ապա UI)։ |
| B3 | Առաջարկ + սպասել | Եթե **մեծ/ադապտիվ** (ճարտարապետություն, նոր գրադարան, ադապտիվ արժեք) | Ներկայացնել **տարբերակներ** + **խորհուրդ**, ասել «Սպասում եմ հաստատում» — **ոչ կոդ** մինչև «ok»/«go»։ |
| B4 | Կոդ գրել | Միայն եթե խնդիրը **պարզ** է կամ **հաստատում** ստացվել է | Անցնել Փուլ C։ Մեծի դեպքում **մի** գրել կոդ մինչև B3-ի հաստատումը։ |

| C3 | Վալիդացիա/սխալներ | Մուտք — Zod/validation, try/catch, error response status + message։ |

### Փուլ D. Ավարտ

| # | Քայլ | Գործողություն |
|---|------|----------------|
| D1 | Build | Կոդ փոխելուց հետո — `pnpm build` չպետք է կոտրվի (կամ նշել, թե ինչու ժամանակավոր անհնար է)։ |
| D2 | Ամփոփում | Գրել «Ինչ արվեց», «Որ ֆայլեր», «Ինչ հաջորդ (եթե կա)»։ |

**Խնդիրներ չառաջացնելու կանոններ.**

- **Մի** գրել կոդ, եթե խնդիրը unclear — նախ հարցնել/պարզաբանել։
- **Մի** ավելացնել նոր թղթապանակ/մեծ ճարտարապետական փոփոխություն առանց 01-ARCHITECTURE-ի և (մեծի դեպքում) հաստատման։
- **Մի** օգտագործել `any`, inline styles, magic numbers, `console.log` production-ում (ըստ 00-core.mdc)։
- **Միշտ** հենվել PROJECT_CONTEXT-ի **սեղմ** կառուցվածքի և API ցանկի վրա — չհնարել ֆայլեր/ճանապարհներ, որոնք doc-ում չկան (կամ նախ առաջարկել և թարմացնել doc)։
